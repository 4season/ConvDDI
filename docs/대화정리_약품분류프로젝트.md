# 약품 분류 프로젝트 — 대화 정리

> CNN 기반 약품 이미지 분류 + 병용금기 검출 (딥러닝 팀 프로젝트, team_project_06)
> 작성일: 2026-06-14 · 사소한 조작/버그 트러블슈팅은 제외하고 의미 있는 결정·결과만 정리

---

## 🎯 프로젝트 개요

- **목표:** 수업에서 배운 CNN 지식을 보여주는 것이 핵심 (정확도 극대화가 목적은 아님)
- **최종 지향점:** 병용금기(drug interaction) 검출 — 개별 약품 이미지를 분류한 뒤, 병용금기 조합 DB(엑셀, 1,628쌍)와 대조
- **데이터:** 약 30GB 약품 이미지 (사진 1장에 약품 4개, 여러 촬영 각도), COCO 포맷 JSON bbox 주석
- **범위 설정:** 100개 약품 클래스 · 128×128 해상도 · 약 80 에폭 (MNIST 같은 토이 데이터보다 의도적으로 더 묵직하게)

---

## 🧱 파이프라인 구조 (PyTorch 4개 파일)

- **`1_preprocess.py`** — bbox 좌표로 다중 약품 사진에서 개별 약품을 크롭 → 약 4만 장대의 단일 약품 이미지 생성
- **`2_model.py`** — DrugCNN: Conv/BN/ReLU/MaxPool 블록 4개 + FC + Dropout + Softmax (약 8.8M 파라미터)
- **`3_train.py`** — CrossEntropyLoss + Adam + early stopping(patience=8) + 손실 곡선 시각화
- **`4_predict_and_check.py`** — CNN 추론 → 약품 코드 → 성분명 → 엑셀 DB에서 병용금기 조회

---

## 🧠 핵심 설계 원칙 & 학습 내용

- **크롭 방식:** bbox 직사각형 크롭만으로 CNN 분류엔 충분. 정밀 외곽선 추출(누끼)은 불필요한 오버헤드.
- **과제 재정의:** 원래 object detection(수업 범위 밖)이었던 문제를, 전처리 단계에서 크롭을 미리 해두는 방식으로 **분류(classification) 문제로 전환**한 것이 핵심 전환점.
- **하이퍼파라미터의 스케일 전이성**
  - 잘 옮겨가는 값: learning rate, batch size, dropout, augmentation
  - 그대로 못 옮기는 값: epoch 수, early stopping 임계, 클래스 수에 민감한 설정
- **클래스 선정 방향:** 병용금기 쌍(1,628개) 표에 자주 등장하는 약을 우선 선택하는 것이 가장 내적으로 일관된 방법.

## 📊 실험 결과

### 데이터 업로드
- Jupyter Lab에 폴더를 그냥 드래그하면 일부(73장)만 올라가는 문제 발생 → **zip으로 묶어 업로드 후 압축 해제**로 해결, 최종 **43,350장** 정상 확보.

### 학습 속도 최적화
- 에폭당 40~50초로 느렸던 원인은 GPU가 아니라 **이미지 파일 I/O 병목**.
- RAM 캐시를 시도했으나 `num_workers=0` 상태에서 CPU transform이 직렬 실행되어 오히려 130초까지 느려짐.
- **`num_workers=4` + `persistent_workers=True`** 로 transform을 병렬화 → **에폭당 3.5초**로 단축. 워커가 데이터를 미리 불러오므로 RAM 캐시는 불필요해져 제거.

### 학습률 비교 (20클래스 · 20에폭, 1단계)

| 항목 | lr=1e-3 | lr=3e-4 |
|---|---|---|
| 최고 val_acc | 99.21% (epoch 17) | **99.37%** (epoch 6·9) |
| 99% 도달 | 느림 | epoch 2 |
| 수렴 안정성 | 보통 | 더 빠름 |

→ **lr=3e-4 채택.** lr=1e-4는 20에폭 기준 과소수렴 우려로 스킵.

### 1차 결과 곡선 분석 (20클래스)
- 클라우드에 matplotlib가 없어 로그 수치로 곡선을 로컬에서 재현.
- best 모델 = **epoch 17** (val_loss 0.0343 최저, 저장 기준이 val_loss임), val_acc 최고점 = 0.9942(epoch 19).
- val_acc가 train_acc보다 계속 높은 건 **버그 아님** — 학습 시 dropout=0.5가 train 지표를 불리하게 만들기 때문. 정상 현상.
- early stopping 미발동(끝까지 개선 중) → 더 배울 여지가 있었다는 의미.

### 100클래스 본 학습 (최종, 2단계)
- **Best val_acc: 99.18%** (epoch 24, val_loss 0.0935)
- early stopping: epoch 32에서 종료, 에폭당 ~13초 × 32 ≈ **약 7분**
- 20클래스(99.37%) 대비 클래스가 5배 늘었음에도 99.18% 유지 — 매우 좋은 결과.

### 최종 모델 검증 (`best_model.pth`)
- 크기 **33.70MB** (8.83M 파라미터 × float32 ≈ 35MB 예상치와 일치), PyTorch 정상 ZIP 포맷.
- 메타 확인: `classes=100`, `val_acc=0.9918`, `epoch=24` — 100클래스 최종 모델 확정.
- 손실 곡선 깔끔, 오버피팅 없이 수렴.
